# Sankofa Payroll Platform Design

**Date:** 2026-03-12
**Project:** Sankofa Payroll & Compliance Services
**Scope:** Internal company web application for payroll operations, compliance review, and audit evidence exploration using fictional seed data.

## Goal

Build a believable internal company dashboard that supports the roles and workflows described in the payroll breach scenario without implementing offensive capability. The application should look and behave like normal business software used by payroll, compliance, and audit staff.

## Product Direction

The platform is a single integrated internal system rather than separate tools. This matches the scenario, where Anita and Felix both operate inside the same trusted company environment and the later audit review depends on records generated by that environment.

The application should feel like a traditional internal corporate portal:

- table-heavy and operational
- business-oriented, not consumer-oriented
- role-aware, but not over-engineered
- realistic enough for a classroom demonstration

## Chosen Stack

- NestJS
- SQLite
- Prisma
- Server-rendered views in the NestJS application
- Session-based authentication suitable for a demo environment

This stack keeps deployment and setup simple while still supporting a real database, role-based routing, and seeded data.

## Roles

### Payroll Admin

Represents Anita-style users. Primary responsibilities:

- manage employee payroll records
- create and review payroll batches
- inspect payroll entries and totals

### Compliance Officer

Represents Felix-style users. Primary responsibilities:

- open quarterly payroll batches
- review policy and compliance checkpoints
- resolve or document flagged records
- sign off on completed reviews

### Audit Analyst

Represents the post-incident investigation role. Primary responsibilities:

- inspect login history
- inspect host and system evidence
- inspect outbound connection records
- review activity trails in read-only mode

## Core Modules

### Authentication

The application uses a simple seeded login flow with demo accounts mapped to the three roles. This is intentionally lightweight and is not meant to be production authentication.

### Dashboard

The landing page is role-aware and surfaces:

- summary metrics
- pending actions
- recent activity
- shortcuts to the role's most relevant areas

### Employees

Provides a searchable employee directory with payroll-relevant details such as department, pay grade, status, and masked banking information.

### Payroll Batches

Handles quarterly payroll runs, including batch status, totals, creator, timestamps, and drill-down into employee payroll entries.

### Compliance Review

Provides Felix-style review workflows around a payroll batch, including checklist completion, flag tracking, comments, and sign-off status.

### Audit & Activity

Provides a read-only investigation area that surfaces mock evidence for the classroom scenario, including:

- login events
- general activity logs
- mock service or host events
- mock outbound network events

This module exists to support the scenario narrative without simulating unauthorized compromise.

## Data Flow

1. A Payroll Admin logs in and creates or views payroll batches.
2. Payroll entries connect batch data back to employee records.
3. A Compliance Officer reviews a batch, inspects flags, and records sign-off decisions.
4. The Dashboard aggregates summaries from payroll, compliance, and activity data.
5. The Audit Analyst later inspects seeded evidence records tied to the same users, devices, and business actions.

This creates a coherent story inside one platform without implementing any harmful behavior.

## Error Handling

- invalid login attempts return a plain internal-system error state
- unauthorized route access redirects to an access denied page
- empty states are explicit and business-oriented
- server-side errors are logged and shown as a generic internal error page
- audit-related records are read-only to preserve seeded evidence during demonstrations

## Seed Data

The app should start with a complete fictional company dataset:

- demo users for payroll, compliance, and audit roles
- 30 to 80 employees across multiple departments
- multiple quarterly payroll batches with different statuses
- realistic payroll entries with salary, tax, and deduction values
- compliance reviews with a mix of passed items and flagged exceptions
- business activity logs
- investigation evidence tied to Felix's workstation and account timeline

All data should be clearly fictional but convincing enough for a classroom walkthrough.

## Testing Strategy

- unit tests for service-level calculations and summaries
- integration tests for login, employee list, payroll detail, compliance review, and audit routes
- seed validation to ensure demo data is reproducible
- role-access tests for all protected sections

## Explicit Non-Goals

The project should not implement:

- weaponized documents
- hidden execution on file open
- persistence or remote-access tooling
- real command-and-control behavior
- real data exfiltration
- live email integration
- production-grade authentication
- real-time notifications

## Recommended Version 1 Scope

Build first:

- login
- role-aware dashboard
- employee directory
- payroll batches and entries
- compliance review area
- audit timeline and evidence tables
- database seed and basic tests

Defer:

- uploads
- exports
- notifications
- advanced analytics
- real-time updates
